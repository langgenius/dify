# ADR 0001: Unify the Dify API gevent startup model

## Status

Accepted

## Context

Before this change, the backend had multiple effective runtime paths:

- The Docker API process could run under a Gunicorn gevent worker, or bypass
  Gunicorn under certain conditions.
- Local development for the API defaulted to `flask run`.
- Local worker / beat entrypoints did not match the Docker worker / beat
  entrypoints.

These differences caused drift in standard library gevent patching,
third-party compatibility patching (for example gRPC and psycopg2), and
debugger behavior. When problems occurred, the visible symptom often appeared
much later in request handling, streaming, or websocket code, which made the
root cause expensive to diagnose.

At the same time, the current codebase already depends on a gevent runtime in
practice, especially in websocket and streaming paths built on
`gevent.pywsgi` and `gevent-websocket`.

## Decision

We standardize on a gevent-only backend startup model:

- The API service always starts through Gunicorn gevent workers.
- `DEBUG` no longer decides whether to switch to a different server runner.
- Local API development also uses Gunicorn gevent workers.
- Worker and beat both enter through `celery_entrypoint`.
- Standard library gevent monkey patching remains owned by framework entrypoints
  such as Gunicorn and Celery.
- Third-party compatibility patching for gRPC and psycopg2 is centralized in a
  dedicated gevent bootstrap helper.
- Supported gevent entrypoints perform fail-fast checks. If required gevent
  patching is missing, the process exits immediately instead of continuing in a
  partially patched state.
- `app.py` remains as a framework integration module, but manual server startup
  through that module is no longer supported.

## Consequences

Positive outcomes:

- Runtime model drift between `DEBUG`, Docker, and local development scripts is
  removed.
- The cost of diagnosing gevent-related deadlocks, blocking behavior, and patch
  ordering issues is reduced.
- API, worker, beat, and debugger behavior become more consistent.

Trade-offs and constraints:

- `flask run` and `python -m app` are no longer treated as supported backend
  entrypoints.
- Gevent bootstrap logic must stay inside a dedicated namespace so unrelated
  code does not leak into startup paths.
- Debugger configurations must explicitly enable gevent compatibility.

## Implementation notes

- The gevent bootstrap helper lives under `api/_dify_gevent_bootstrap/`.
- That directory is reserved for gevent patching and startup checks only. No
  unrelated Python code should be added there.
- VS Code / Cursor debug templates must enable gevent-related debugger options.
- When debugging the backend in IntelliJ IDEA or PyCharm, the `"Gevent compatible"`
  option must be enabled.
