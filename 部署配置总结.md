## 二、代码拉取

### 2.1 首次拉取（失败）
```bash
git clone https://github.com/langgenius/dify.git /home/project/dify
```
**问题**: RPC 失败，curl 56 Recv failure，连接超时
```
error: RPC 失败。curl 56 Recv failure: 连接超时
error: fetch-pack: unexpected disconnect while reading sideband packet
fatal: 过早的文件结束符（EOF）
```

### 2.2 解决方案
使用浅克隆（shallow clone）减少数据传输：
```bash
git clone --depth 1 https://github.com/langgenius/dify.git /home/project/dify
```
**结果**: 成功，下载了 28.49 MiB 数据

## 四、后续建议

```bash
SMTP_PORT=587
SMTP_USERNAME=your_email@qq.com
SMTP_PASSWORD=your授权码
SMTP_USE_TLS=true
```

如需更新 Dify 版本：
```bash

docker compose up -d
```

## 五、常见问题与解决方案

### 5.1 模型配置错误

#### 错误信息
```
1 validation error for RuleGeneratePayload
model_config.mode
Input should be 'completion' or 'chat' [type=enum, input_value='agent-chat', input_type=str]
```

**原因**:
- `agent-chat` 是 Dify 的**应用模式**（AppMode），不是 **LLM 模式**
- 后端 `RuleGeneratePayload` 的 `ModelConfig.mode` 只接受 `chat` 或 `completion`
- Agent 应用底层调用 LLM 时应使用 `chat` 模式

**说明**: 这是 Dify 代码的问题，在使用 Agent 模式的应用进行提示词生成时会触发。如果需要修复，需要在前端代码中将 `agent-chat` 映射为 `chat`。

### 5.2 模型连接错误

#### 错误信息
```
An error occurred during credentials validation:
HTTPConnectionPool(host='localhost', port=11436):
Max retries exceeded with url: /v1/chat/completions
Connection refused
```

**原因**: 尝试连接本地 Ollama 服务但连接被拒绝

**解决**: 确保 Ollama 服务正在运行
```bash
ollama serve
ollama pull <model-name>
```

### 5.3 模型类型不匹配

#### 错误信息
```
Credentials validation failed with status code 404
{"detail":"Not Found"}
```

**原因**:
- 使用了 `Qwen3-Embedding-8B`（Embedding 模型）作为 LLM 使用
- Embedding 模型不能用于对话生成

**说明**:
| 模型类型 | 用途 | 示例 |
|---------|------|------|
| LLM | 对话生成 | Qwen2.5-7B-Instruct |
| Embedding | 文本向量化 | Qwen3-Embedding-8B |

### 5.4 通知接口 404 错误

#### 错误信息
```
/console/api/notification 返回 404
```

**原因**:
- `EDITION=CLOUD` 时，Dify 会调用外部 BILLING_API_URL
- 自托管环境无法访问 Dify 官方服务器

**说明**: 这是预期行为，不影响核心功能。可忽略或配置 `EDITION=SELF_HOSTED`

### 5.5 用户注册需要邮箱验证码

#### 问题
开启了 `ENABLE_EMAIL_CODE_LOGIN=true`，但未配置 SMTP 邮件服务

**解决**:
```bash
# 关闭邮箱验证码，仅使用密码登录
ENABLE_EMAIL_CODE_LOGIN=false
ENABLE_EMAIL_PASSWORD_LOGIN=true
```

### 6.2 常用 Git 操作

#### 查看远程仓库
```bash
# 查看所有远程仓库
git remote -v

# 查看远程仓库详情
git remote show origin

# 查看分支追踪状态
git branch -vv
```

#### 添加远程仓库
```bash
# 添加新的远程仓库
git remote add <名称> <仓库地址>

# 示例
git remote add AI-dify-dev git@github.com:CarlosJoe/AI-dify-dev.git
```

#### 删除远程仓库
```bash
git remote remove <名称>
# 示例
git remote remove origin
```

#### 重命名远程仓库
```bash
git remote rename <旧名称> <新名称>
# 示例
git remote rename AI-dify-dev origin
```

#### 设置分支追踪
```bash
# 设置本地分支追踪远程分支
git branch --set-upstream-to=<远程>/<分支> <本地分支>
# 示例
git branch --set-upstream-to=origin/main main
```

#### 拉取代码
```bash
# 拉取并合并（默认模式）
git pull origin main

# 拉取并变基
git pull --rebase origin main

# 仅拉取（不合并）
git fetch origin main
```

#### 推送代码
```bash
# 推送到远程仓库
git push origin main

# 首次推送并设置追踪
git push -u origin main
```

#### 处理分支分叉

当本地分支与远程分支分叉时（本地领先或落后于远程），Git 会提示需要指定如何调和：

```bash
# 方式一：合并（保留完整历史）
git config pull.rebase false
git pull origin main

# 方式二：变基（历史更整洁）
git config pull.rebase true
git pull origin main

# 方式三：仅快进
git config pull.ff only
git pull origin main
```

#### 同步主仓库更新（Fork 仓库）

当 fork 仓库需要同步主仓库的最新代码时：

```bash
# 1. 添加主仓库为 upstream（只需执行一次）
git remote add upstream https://github.com/langgenius/dify.git

# 2. 拉取主仓库最新代码
git fetch upstream main

# 3. 合并到本地分支
git merge upstream/main

# 4. 解决冲突后，推送到自己的 fork 仓库
git push origin main
```

> **注意**：合并冲突需要手动解决。常见的冲突场景包括：
> - `docker/docker-compose.yaml`：上游使用预构建镜像，本地可能使用本地构建
> - `api/Dockerfile`：上游可能新增依赖或变更构建配置

| 日期 | 合并提交 | 说明 |
|------|---------|------|

### 6.3 忽略文件权限变化

如果遇到大量文件被标记为修改（仅权限变化），可以关闭文件权限检查：

```bash
# 关闭文件权限追踪
git config core.filemode false

# 查看当前配置
git config --list | grep filemode
```
