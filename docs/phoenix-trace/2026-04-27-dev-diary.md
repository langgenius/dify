# Phoenix Trace Feature 开发日记

日期：2026-04-27
基线提交：`949f93069881162e188c282574ac4cadc7db2490`
目标：让 Dify 的 Phoenix tracing 正确表达 workflow、chatflow、nested workflow 和 workflow-as-tool 的 trace hierarchy、session 继承与父 span 关系。

## Feature 目的

这个 feature 解决 Phoenix UI 中 workflow trace 展示不完整的问题：顶层 workflow 容易形成 orphan root，nested workflow 会被拆到独立 session，workflow-as-tool 触发的 child workflow 也无法挂回外层 tool span。

实现需要兼容 Dify 现有 tracing contract：上游仍负责 trace payload 生成、workflow 执行上下文和 task dispatch，Phoenix provider 只补齐 Phoenix/OpenTelemetry 显示所需的 root、session、span hierarchy 和 provider-local parent span coordination。最终修复还把 pending retry contract 收回到 `core.ops`，避免通用 task 直接依赖 Phoenix provider symbol。

## 实现与迭代过程

第一阶段从 Phoenix session 页面 `rootSpan: null` 和 workflow span orphan-root 问题切入。分析后确认问题不是单个字段缺失，而是 root context、`session.id` 和 workflow 内部 span parent 规则没有形成一致语义。决策记录将 v1 范围收敛到 Phoenix provider：canonical root 必须稳定，workflow node hierarchy 暂由 Phoenix local helper 重建，未来如果上游 tracing builder 能提供统一 hierarchy，再迁出 provider。

第二阶段先用测试冻结 Phoenix root、session fallback、message session 和 node hierarchy 预期，然后在 provider 内补 helper：workflow root span 使用真实 root context，session fallback 走 `conversation_id`、父 workflow run id、当前 workflow run id，node span 则按显式 predecessor、graph parent、structured parent、workflow root 逐级 fallback。这个阶段也处理了 repeated nodes 和 loop/iteration body node，避免相同 graph node 多次执行时覆盖 parent 关系。

第三阶段发现 workflow-as-tool 的 nested workflow 缺少 `parent_trace_context`，于是把工作从 Phoenix provider 扩展到 Dify backend 调用链。`trace_id_helper` 负责提取 parent context，workflow tool runtime 通过内部 binding 携带外层 workflow run id 和 node execution id，`WorkflowAppGenerator.generate()` 与 workflow persistence 再把 context 写入 trace task metadata。这个迭代中曾尝试支持 editable graphon model runtime imports，随后撤回，保留与当前 graphon v0.2.2 fixture 对齐的测试调整。

第四阶段解决 nested workflow 已进入同一 trace/session 但仍挂不到外层 tool span 的问题。因为 Phoenix span 由异步 task 发送，child workflow task 可能早于 parent tool span context 可用，所以 provider 通过 Redis 发布 tool node span carrier；child workflow 如果还拿不到父 carrier，会抛出 core 层 `PendingTraceParentContextError`，由 Celery 做 bounded retry。这个协调逻辑目前保持 Phoenix-local，并在代码注释中记录了未来抽象到 core storage/retry contract 的条件。

第五阶段来自 feature review。审查发现内部 trace metadata 曾经通过公开 `runtime_parameters` 传递，可能覆盖用户 workflow-as-tool 输入；通用 `ops_trace_task` 也曾直接 import Phoenix provider 异常；pending-parent retry 还可能重复发送 enterprise trace。修复后 parent context 走私有 workflow tool binding，通用 task 只依赖 `RetryableTraceDispatchError`，并在 retry payload 中持久化 enterprise dispatch 状态，避免 retry 重复发送。

最后一次整理把两个同名 `test_arize_phoenix_trace.py` 合并为一个 provider 测试入口。两个文件历史上都存在，但本 feature 同时扩展两边后会让 pytest 节点和 review diff 变得混乱；合并后保留全部覆盖，删除外层重复命名文件。

## 本地 Feature 测试覆盖点

- Phoenix provider root/session：canonical root、workflow run root name、nested workflow session inheritance、conversation session 保留。
- Phoenix workflow hierarchy：serial predecessor、graph parent fallback、structured loop/iteration parent、parallel-like ambiguous fallback、graph title 优先。
- Workflow-as-tool parent context：外层 workflow run id 与 node execution id 进入 nested workflow metadata，同时不覆盖用户同名 workflow 输入。
- Pending parent retry：Phoenix parent span carrier 缺失时抛出 core retry exception，trace task 保留 payload 并做 bounded retry。
- Enterprise trace 幂等：pending retry 后跳过已成功发送的 enterprise dispatch。
- Trace dispatch 边界：`ops_trace_task` 依赖 provider-neutral retry contract，不直接依赖 Phoenix provider。
- 测试整理：Phoenix provider helper tests 与 integration-like provider tests 合并到单一 `arize_phoenix_trace/test_arize_phoenix_trace.py`。
