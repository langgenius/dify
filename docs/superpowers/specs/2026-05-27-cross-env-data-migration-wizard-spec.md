# Cross-Environment Data Migration Wizard Spec

## Purpose

Turn the current cross-environment custom data migration CLI into a reusable product-grade export/import workflow.

The current branch can export and import workflows, custom API tools, workflow tools, and MCP tools, but its behavior is closer to a one-off enterprise migration script. This spec defines a more general model centered on explicit tenant selection, safer secret handling, dependency discovery, and an interactive export wizard.

## Goals

- Export workflow and chatflow apps with their referenced custom tools.
- Make export safe by default by not exporting secrets unless explicitly requested.
- Avoid overloading `tenant_name` as both source and target tenant identity.
- Support interactive export through a wizard that directly writes the final output JSON.
- Add the wizard as incremental functionality without breaking existing Dify CLI behavior.
- Keep scripted export/import paths possible and consistent with wizard behavior.
- Make source and target tenant semantics explicit in intermediate JSON files.
- Move reusable migration logic into a service layer that can be reused by Flask CLI commands and future Dify CLI entrypoints.
- Produce clear reports showing exported, skipped, and dependency-only resources.

## Non-Goals

- Do not build a UI page for export/import in this iteration.
- Do not serialize built-in or plugin tool implementations into the migration package.
- Do not solve cross-environment plugin installation automatically.
- Do not require a separate persisted export plan file.
- Do not make UUID preservation the only supported import strategy.
- Do not support multi-source-tenant wizard export in the first version.
- Do not support per-app target tenant mapping.

## Terminology

- **Source tenant**: the tenant/workspace resources are exported from.
- **Target tenant**: the tenant/workspace resources are imported into.
- **Migration package**: the final output JSON generated by export and consumed by import.
- **Referenced tools**: tools discovered from selected workflow/chatflow app graphs.
- **Dependency metadata**: non-serializable dependencies, such as built-in/plugin tool references, recorded so operators know what must exist in the target environment.

## Commands

The current branch-only names `export-custom-data` and `import-custom-data` are vague. Since this feature has not shipped yet, they can be renamed before release. Prefer naming around data migration, for example:

- `export-migration-data`
- `import-migration-data`
- `migration-data-wizard`

Release behavior:

- Ship only the migration-data command names.
- Do not add legacy wrappers for the branch-only `export-custom-data`, `import-custom-data`, or `migration-custom-datas` names.
- Import should require the new migration package format with `metadata.version`.
- If a package has no `metadata.version`, fail with an actionable format error instead of trying to interpret branch-only draft data.

CLI commands should stay thin. They should parse arguments, run wizard prompts, call migration services, and print reports. They should not contain resource-specific export/import business logic.

## Service Layer

Reusable migration logic should live outside `api/commands/system.py` in a dedicated service layer.

Suggested module boundary:

```text
api/services/data_migration/
  entities.py
  export_service.py
  import_service.py
  dependency_discovery_service.py
  package_service.py
  report_service.py
```

Responsibilities:

- `export_service.py`: build migration package data from an export selection.
- `import_service.py`: apply a migration package to a resolved target tenant using explicit strategies.
- `dependency_discovery_service.py`: scan workflow/chatflow app graphs and classify referenced tools.
- `package_service.py`: validate package version, load/save JSON, and reject unsupported draft package formats with actionable errors.
- `report_service.py`: generate structured export/import reports for CLI output and future integrations.
- `entities.py`: define typed request/result objects such as export selection, import options, dependency records, and report models.

Design requirements:

- Flask CLI commands and future `dify-cli` commands should call the same service APIs.
- Service APIs should accept structured inputs and return structured results instead of printing directly.
- Click-specific prompting and console formatting should remain in command/wizard adapters.
- Import/export services should be testable without invoking Click commands.
- The service layer should leave room for a future standalone Dify CLI entrypoint without duplicating migration logic.

### Interactive Export

Add a guided command:

```bash
flask migration-data-wizard
```

The wizard asks questions, keeps the selection in memory, confirms the final selection, then directly writes the migration package JSON.

It must not create or require a separate plan file.

### Scripted Export

Keep a scripted export path for automation:

```bash
flask export-migration-data --input export-config.json --output migration-package.json
```

The scripted path should reuse the same export service as the wizard.

### Scripted Import

Update import so target tenant is explicit:

```bash
flask import-migration-data --input migration-package.json --target-tenant prod-workspace
```

Import should not require source and target tenant names to match.

## Migration Package Shape

The export package should distinguish metadata from import target selection.

Both source and target tenant semantics should be represented in intermediate files:

- Export config input should identify the source tenant scope.
- Migration package output should record source tenant metadata.
- Import config input, if used, should identify the target tenant scope.
- The migration package may record intended target tenant metadata when the operator provides it, but import must still allow overriding target tenant from CLI.

Example:

```json
{
  "metadata": {
    "version": "1",
    "source_scope": "single",
    "source_tenants": [
      {
        "id": "source-tenant-id",
        "name": "source-workspace"
      }
    ],
    "target_tenant": null,
    "created_at": "2026-05-27T12:00:00Z",
    "include_secrets": false,
    "import_options": {
      "create_app_api_token_on_import": false,
      "id_strategy": "preserve-id",
      "conflict_strategy": "fail"
    }
  },
  "workflows": [],
  "tools": [],
  "workflow_tools": [],
  "mcp_tools": [],
  "dependencies": []
}
```

Rules:

- `metadata.source_tenants` is informational.
- `metadata.source_scope` is `single` in the first version.
- `metadata.target_tenant` is optional intended target metadata, not a hard requirement.
- Target tenant can be provided at import time and should override package metadata.
- Import target precedence is: CLI `--target-tenant` first, import config `target_tenant` second, package `metadata.target_tenant` third.
- If no target tenant can be resolved from those sources, import fails before creating or updating resources.
- `metadata.import_options` stores import-time behavior requested during export or wizard flow. CLI/import config options may override these values during import.
- Resource records should not rely on `tenant_name` for target lookup.
- Package version should be present to support future format changes.

## Export Wizard Flow

1. Ask which tenant to export from.
   - Show available tenants.
   - First version supports exactly one source tenant.
   - Do not include an `all` option in the wizard v1.

2. Ask which apps to export.
   - First version supports workflow and advanced-chat apps only.
   - Show app name, mode, and ID.
   - Support selecting all eligible apps for the selected tenant.

3. Ask whether to automatically export tools referenced by selected apps.
   - Default should be yes.
   - If yes, parse selected app graphs and discover dependencies.

4. Discover referenced tools.
   - Scan standalone tool nodes.
   - Scan agent node tool configs.
   - De-duplicate discovered resources.
   - Classify discovered resources by type.

5. Ask whether additional tools should be exported manually.
   - If yes, show categorized tool lists.
   - Automatically discovered tools should be preselected.

6. Show categorized tool selection.
   - Discovered workflow tools.
   - Discovered custom API tools.
   - Discovered MCP tools.
   - Other workflow tools in scope.
   - Other custom API tools in scope.
   - Other MCP tools in scope.
   - Built-in/plugin tools discovered as dependency metadata only.

7. Ask for secret export policy.
   - Default: do not export secrets.
   - If secrets are included, show a warning that the output JSON is sensitive.

8. Ask whether imported apps should create or reuse app API tokens.
   - Default: no.
   - Store the choice as `create_app_api_token_on_import`.
   - Make clear that this controls app API token creation during import, not workflow publication.

9. Ask for output path.
   - Provide a default path.
   - Suggested default: `migration-data-YYYYMMDD-HHMMSS.json`.

10. Show final confirmation summary.
   - Tenants selected.
   - Apps selected.
   - Tools selected by category.
   - Dependency-only resources.
   - Secret export policy.
   - App API token creation policy.
   - Output path.

11. Write the migration package JSON and print the export report.

## Dependency Discovery

Dependency discovery should produce typed results:

- `workflow_tools`: export the `WorkflowToolProvider` and ensure the workflow app referenced by the workflow tool is included or reported.
- `api_tools`: export custom API tool provider schema and metadata. Credential inclusion follows the secret policy.
- `mcp_tools`: export MCP provider data. URL, headers, and authentication inclusion follow the secret policy.
- `builtin_or_plugin_tools`: record dependency metadata only.

Workflow tool app dependency behavior:

- App migration is the primary flow; tool migration is supporting data.
- The exporter should not auto-include workflow apps just because a manually selected workflow tool references them.
- Workflow tools discovered from selected apps should be exported as supporting dependencies.
- Workflow tools manually selected outside the selected app set should be exported as independent resources.
- Manually selected workflow tools should always be treated as not referenced by selected apps, even if their underlying app IDs happen to overlap with selected apps.
- Workflow tools may be migrated even if no selected app currently references them.
- On import, workflow tools should be allowed to create/update independently when their referenced workflow app exists in the target environment or can be resolved through the import ID mapping.
- If the referenced workflow app cannot be resolved, import should not silently create a broken workflow tool. It should follow the configured conflict/unresolved-reference strategy. The conservative default is to skip that workflow tool and report it as unresolved.

MCP export behavior:

- In the secret-free default mode, referenced MCP tools should be recorded as dependency metadata only.
- MCP providers should be fully exportable only when secret export is explicitly enabled, because server URL, headers, and authentication often contain deploy-specific or sensitive values.
- Import should skip dependency-only MCP entries and report that the target environment requires manual MCP provider setup.

The report should include:

- discovered count by type,
- exported count by type,
- skipped count by type,
- dependency-only items that must be prepared in the target environment.

## Secret Policy

Export must be safe by default.

Default behavior:

- API tool credentials are not exported.
- Workflow secret variables are not exported.
- Tool/agent node credential IDs are removed or marked as unresolved dependencies.
- MCP providers are recorded as dependency metadata only unless secrets are explicitly included.

Explicit secret export:

- Provide an option equivalent to `--include-secrets`.
- The wizard must show a warning before enabling it.
- The migration package metadata must record `include_secrets: true`.

## Import Semantics

Import should take target tenant from CLI options, not from resource records:

```bash
flask import-migration-data --input migration-package.json --target-tenant prod-workspace
```

The current implementation resolves tenant per resource from `tenant_name`. If the tenant is missing, that resource fails and the importer continues with the next resource. The new import flow should not keep that behavior for target tenant resolution.

Target tenant must be resolved once before import starts. If the target tenant cannot be found, import should fail fast before creating or updating any resource.

Operator identity should be explicit:

- Prefer an `--operator` or `--operator-email` option.
- If omitted, use a well-defined fallback and report it.
- Avoid silently choosing the earliest owner account.

ID and conflict strategy should be explicit:

- ID strategy examples: `preserve-id`, `generate-new-id`, `map-id`.
- Conflict strategy examples: `fail`, `skip`, `update`, `replace`.
- New import default ID strategy is `preserve-id`, matching the migration package's reference-preservation goal.
- New import default conflict strategy is `fail`, so imports do not overwrite target resources unless the operator explicitly chooses `update` or `replace`.

API token creation should be opt-in and clearly named.

Replace misleading `workflow_publish_api` semantics with a name that describes actual behavior, such as:

```json
"create_app_api_token_on_import": true
```

This controls app API token creation/reuse during import. It does not control workflow publication.

## Scripted Export Config

The current scripted input config can evolve toward explicit names:

```json
{
  "source_tenant": {
    "mode": "single",
    "name": "admin's Workspace"
  },
  "apps": {
    "modes": ["workflow", "advanced-chat"],
    "ids": [],
    "all": true
  },
  "include_referenced_tools": true,
  "additional_tools": {
    "api_tools": [],
    "workflow_tools": [],
    "mcp_tools": []
  },
  "include_secrets": false,
  "import_options": {
    "create_app_api_token_on_import": false,
    "id_strategy": "preserve-id",
    "conflict_strategy": "fail"
  }
}
```

This keeps scripted export close to the wizard choices while avoiding ambiguous `tenant_name` semantics.

The first wizard version is intentionally single-tenant only. Multi-source-tenant export can be revisited later for scripted backup/export use cases, but it should not introduce per-app target tenant mapping in this iteration.

If a scripted import config is introduced, it should make target tenant explicit:

```json
{
  "input": "migration-package.json",
  "target_tenant": {
    "name": "prod-workspace"
  },
  "operator_email": "admin@example.com",
  "id_strategy": "preserve-id",
  "conflict_strategy": "update"
}
```

## Error Handling

- If the import target tenant is missing, fail before creating or updating any resource.
- If a selected tenant is missing, fail before writing output.
- If selected app IDs are missing, report missing IDs and fail unless a skip policy is explicitly provided.
- If dependency discovery cannot resolve a referenced tool, record it as unresolved and report it.
- If the output path exists, ask before overwriting in wizard mode.
- In scripted mode, require an explicit overwrite flag or fail.

## Reporting

Export report should include:

- output path,
- source tenant scope,
- selected app count,
- exported workflow count,
- exported API tool count,
- exported workflow tool count,
- exported MCP tool count,
- dependency-only built-in/plugin tool count,
- unresolved dependency count,
- whether secrets were included.

Import report should include:

- target tenant,
- operator account,
- created/updated/skipped count by resource type,
- generated ID mapping if applicable,
- created/reused app API token count,
- unresolved dependencies.

## Testing

Add focused tests for:

- tenant-scoped `export_all_workflows`,
- wizard selection to export selection conversion,
- dependency discovery from standalone tool nodes,
- dependency discovery from agent node tools,
- de-duplication across multiple apps,
- secret exclusion by default,
- secret inclusion only when explicitly requested,
- import target tenant override,
- conflict and ID strategy behavior.

## Open Decisions

- Exact CLI option names for operator identity, ID strategy, conflict strategy, and overwrite behavior.
