# Weaviate Migration Guide: v1.27 → v1.38

## Overview

Dify's bundled self-hosted Weaviate is upgraded from **`1.27.0`** to **`1.38.2`**
(the latest `1.38` patch), and the Python `weaviate-client` from **`4.20.5`** to
**`4.22.0`** (see [PR #38214](https://github.com/langgenius/dify/pull/38214)).

The Dify vector-store code path is unchanged: collections keep the layout
`Vector_index_<dataset_id>_Node` with a self-provided named vector `default`
(cosine distance) and the same property set (`text`, `document_id`, `doc_id`,
`doc_type`, `chunk_index`, `is_summary`, `original_chunk_id`). Existing data is
**forward compatible** — no re-indexing is required.

This is a large jump (11 minor releases). Weaviate does **not** test or support
skipping minor versions, so the supported path is to upgrade **one minor version
at a time**. This guide gives the exact ladder and two operational details that
matter in practice (graceful shutdown, and post-restart readiness).

> **TL;DR**
> 1. **Back up** the `weaviate` volume first.
> 2. Upgrade **one minor at a time**, using the **latest patch** of each minor:
>    `1.27 → 1.28 → 1.29 → … → 1.38`.
> 3. **Stop Weaviate gracefully** (`docker compose stop` / `down`, never a hard
>    kill) between steps so it flushes the vector index.
> 4. After each restart, **wait a few seconds** for the class index to mount
>    before relying on search.

---

## The golden rule: one minor version at a time

Upgrade sequentially through every minor release, always landing on that minor's
**latest patch**. Do not skip minors.

This mirrors [Weaviate's official upgrade guidance](https://docs.weaviate.io/deploy/migration):

> "Plan to upgrade one minor version at a time, always using the latest patch
> version of each minor release." … this "helps to minimize the risk of issues
> during the upgrade process, by mirroring our testing and release process."

**Recommended ladder from Dify's `1.27` to `1.38`** (latest patch of each minor
at the time of writing — always double-check for a newer patch):

| Step | Image tag |
|------|-----------|
| 0 (current) | `cr.weaviate.io/semitechnologies/weaviate:1.27.x` |
| 1 | `cr.weaviate.io/semitechnologies/weaviate:1.27.27` |
| 2 | `cr.weaviate.io/semitechnologies/weaviate:1.28.16` |
| 3 | `cr.weaviate.io/semitechnologies/weaviate:1.29.11` |
| 4 | `cr.weaviate.io/semitechnologies/weaviate:1.30.23` |
| 5 | `cr.weaviate.io/semitechnologies/weaviate:1.31.22` |
| 6 | `cr.weaviate.io/semitechnologies/weaviate:1.32.27` |
| 7 | `cr.weaviate.io/semitechnologies/weaviate:1.33.18` |
| 8 | `cr.weaviate.io/semitechnologies/weaviate:1.34.20` |
| 9 | `cr.weaviate.io/semitechnologies/weaviate:1.35.23` |
| 10 | `cr.weaviate.io/semitechnologies/weaviate:1.36.21` |
| 11 | `cr.weaviate.io/semitechnologies/weaviate:1.37.12` |
| 12 | `cr.weaviate.io/semitechnologies/weaviate:1.38.2` (Dify default) |

> Dify pins `1.38.2` (the latest `1.38` patch) as the default. Always check for a
> newer `1.38.x` patch and prefer it as the final rung.

> **Always run the latest patch — and stay near the newest minor.** Patch
> releases (e.g. `1.38.latest`) carry the most relevant bug and security fixes.
> Weaviate backports fixes to the latest patch of the **newest minor and roughly
> the three minors before it** — so a version that is more than ~3 minors behind
> the newest release generally stops receiving those backported fixes. Two
> takeaways: (1) once you land on a minor, immediately move to its **latest
> patch**; and (2) don't let your deployment drift far behind — keeping within
> ~3 minors of the newest release is what keeps you in the fix/backport window.
> This applies after the migration too: revisit periodically and bump to the
> current `1.38.x` (or newer) patch.

**Why not just jump straight to 1.38?** In our controlled test the direct jump
happened to preserve the data too (see [Test evidence](#test-evidence)), but
skipping versions is untested upstream: any single release may introduce an
on-disk migration that expects the previous version to have run at least once.
Stepping one minor at a time also means that if something does go wrong you roll
back **one** version instead of debugging an 11-version jump.

## Before you start: back up

Always back up before a production upgrade.

**Simplest (Docker, offline copy of the volume):**

```bash
cd docker
docker compose stop weaviate                       # graceful — see note below
sudo cp -r ./volumes/weaviate ./volumes/weaviate_backup_$(date +%Y%m%d)
```

**Production (recommended): use Weaviate's backup module** to S3/GCS/Azure so you
have a restorable snapshot rather than a raw file copy
([backup docs](https://docs.weaviate.io/deploy/configuration/backups)). The local
filesystem backend is for development only.

## Upgrade steps — Docker users (the common Dify deployment)

Dify pins the Weaviate image in three files:

- `docker/docker-compose.yaml`
- `docker/docker-compose.middleware.yaml`
- `docker/docker-compose-template.yaml`

For each rung in the [ladder](#the-golden-rule-one-minor-version-at-a-time),
repeat this loop:

```bash
cd docker

# 1. Edit the image tag to the next minor's latest patch, e.g.:
#      image: cr.weaviate.io/semitechnologies/weaviate:1.28.16
#    (update all three compose files, or just the one you deploy with)

# 2. Stop Weaviate GRACEFULLY so it flushes its indexes (critical — see below)
docker compose stop weaviate

# 3. Start the new version
docker compose up -d weaviate

# 4. Wait for it to come up, then confirm the reported version
sleep 10
curl -s -H "Authorization: Bearer ${WEAVIATE_API_KEY:-WVF5YThaHlkYwhGUSmCRgsX3tD5ngdN8pkih}" \
  http://localhost:8080/v1/meta | python3 -c "import sys,json;print(json.load(sys.stdin)['version'])"

# 5. Verify data + search (see "Verify after each step"), THEN move to the next rung.
```

Do **not** jump the tag straight to `1.38.2`. Walk the ladder, verifying at each rung.

> `docker compose down` (without `-v`) is also graceful and preserves the named
> volume — but `down` stops the **whole** stack. `docker compose stop weaviate`
> keeps the rest of Dify running while you step Weaviate.

## Upgrade steps — source / middleware installs

If you run Weaviate via `docker-compose.middleware.yaml` while developing:

```bash
cd docker
# For each rung: edit the weaviate image tag in docker-compose.middleware.yaml, then:
docker compose -f docker-compose.middleware.yaml stop weaviate
docker compose -f docker-compose.middleware.yaml --profile weaviate up -d weaviate
sleep 10
curl -s http://localhost:8080/v1/meta | python3 -c "import sys,json;print(json.load(sys.stdin)['version'])"
```

Update the client once (it is compatible with every server 1.27–1.38):

```bash
cd api
uv sync
uv run --project . python -c "import weaviate; print(weaviate.__version__)"   # 4.22.0
```

## Two things that actually matter in practice

### 1. Stop Weaviate **gracefully** before every upgrade

This is the single most important operational detail, and it is easy to get
wrong.

Weaviate keeps its HNSW vector index in memory and flushes it to disk on a clean
shutdown. If the container is **hard-killed** (SIGKILL — e.g. `docker kill`,
`docker rm -f`, an OOM kill, or a `stop` that exceeds the grace period), the
vector-index commit log on disk can be **incomplete**. On the next start:

- The **object store is durable** — every object and its vector survive
  (`aggregate` count is correct, fetching an object by id returns its exact
  vector), **but**
- the **HNSW graph is truncated** — `near_vector` search recalls only a subset of
  objects until the index is rebuilt.

In our testing, a hard kill between versions left `near_vector` returning only
3/10 objects while the object count still read 10. A **graceful stop** (SIGTERM
with enough time to flush) preserved the vector index completely.

**What to do:**
- Use `docker compose stop` / `docker compose down` (both send SIGTERM), never
  `docker kill` / `docker rm -f`.
- Give it time to flush. Raise Docker's stop grace period for large indexes:
  ```bash
  docker compose stop -t 120 weaviate
  ```
- Dify's compose already sets `restart: always`, so normal restarts are graceful;
  the risk is manual hard kills and host OOM.

### 2. Wait for the class index to mount after a restart

On newer versions (observed from ~1.33+), the HTTP endpoint (`/v1/meta`) reports
ready a moment **before** the per-collection index is mounted. A query fired in
that window can fail with:

```
tried to browse non-existing index for Vector_index_<id>_Node
```

This is transient. Wait/retry for a few seconds after each restart before
assuming search is available. Dify's client already tolerates brief startup races
(`skip_init_checks=True` + retries), but if you script health checks, poll until a
trivial query succeeds rather than trusting `/v1/meta` alone.

## Verify after each step

Confirm both data integrity **and** that search works — not just that the server
booted:

```bash
KEY="${WEAVIATE_API_KEY:-WVF5YThaHlkYwhGUSmCRgsX3tD5ngdN8pkih}"

# Reported server version
curl -s -H "Authorization: Bearer $KEY" http://localhost:8080/v1/meta \
  | python3 -c "import sys,json;print('version', json.load(sys.stdin)['version'])"

# Collections are present
curl -s -H "Authorization: Bearer $KEY" http://localhost:8080/v1/schema \
  | python3 -c "import sys,json;print([c['class'] for c in json.load(sys.stdin)['classes']])"

# Object count for a collection (replace <Class>)
curl -s -H "Authorization: Bearer $KEY" \
  "http://localhost:8080/v1/objects?class=<Class>&limit=1" \
  | python3 -c "import sys,json;d=json.load(sys.stdin);print('totalResults', d.get('totalResults'))"
```

Then, in the Dify UI: open a Knowledge base, run a test search, and confirm
retrieval returns chunks. Only proceed to the next rung once search works.

## Rollback

If a rung misbehaves, roll back **one** version:

```bash
cd docker
docker compose stop weaviate
# restore the pre-upgrade backup
sudo rm -rf ./volumes/weaviate
sudo cp -r ./volumes/weaviate_backup_YYYYMMDD ./volumes/weaviate
# set the image tag back to the last known-good rung, then:
docker compose up -d weaviate
```

Because you upgraded one minor at a time, the previous known-good version is only
one step back.

## Test evidence

This guide was validated end-to-end against a Weaviate instance configured exactly
like Dify's `weaviate` service (API-key auth, `DEFAULT_VECTORIZER_MODULE=none`,
ports 8080/50051), seeded on **1.27.0** with a Dify-shaped collection
(`Vector_index_<id>_Node`, self-provided named vector `default`, cosine, Dify's
property set) holding 10 objects with BYO vectors.

**Incremental ladder — data survived every step:**

```
1.27.0 (seed) → 1.27.27 → 1.28.16 → 1.29.11 → 1.30.23 → 1.31.22 → 1.32.27
             → 1.33.18 → 1.34.20 → 1.35.23 → 1.36.21 → 1.37.12 → 1.38.2   ✅
```

At **every** rung, on the same data volume, all of the following held:

- object **count preserved** (10/10),
- **data integrity**: the probe object fetched by UUID kept its exact properties
  **and** its exact stored vector (byte-for-byte, within float32 tolerance),
- **vector search** (`near_vector` on the `default` named vector) returned all
  objects and recalled the probe,
- **BM25** full-text search returned the expected chunk.

**Direct jump 1.27.0 → 1.38.2 (skipping intermediates):** also preserved the data
in this controlled test — but this is *not* the supported path; the incremental
ladder above is what we recommend and what Weaviate officially supports.

**Key finding:** the decisive factor for a clean upgrade was **graceful
shutdown**. Hard-killing the container between versions left the HNSW index
incomplete on reload (objects survived, but `near_vector` recall dropped) until a
rebuild; a graceful SIGTERM stop preserved the vector index fully. See
[Two things that actually matter](#two-things-that-actually-matter-in-practice).
