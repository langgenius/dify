# ADR-0001: UUID v7 for All Primary Keys

**Status:** Accepted
**Date:** 2026-05-19
**Author:** Narayana Chavva
**Applies to:** All tables in the Credits Platform schema (ENG-4 onwards)

---

## Context

The Credits Platform schema is insert-heavy: `credit_ledger`, `usage_events`, and `rating_decisions` each receive a new row for every workflow run. Primary key strategy directly affects index locality and insert throughput.

Three realistic options:

| Strategy | Ordering | Client-side generation | Notes |
|---|---|---|---|
| Auto-increment `BIGINT` | Sequential | No — DB assigns | Leaks row counts to callers; requires round-trip to DB for every INSERT before FK references can be built |
| UUID v4 | Random | Yes | Random distribution causes B-tree page splits on high-volume inserts; index bloat over time |
| **UUID v7** | Time-ordered | Yes | 48-bit millisecond timestamp prefix; new rows cluster at the end of the index |

The platform also uses `idempotency_key` on ledger entries and `event_id` on usage events. These keys must be generated by the service before the DB insert so the caller can safely retry without duplicating records. That requires client-side generation — ruling out auto-increment.

---

## Decision

**All primary keys are UUID v7, generated at the application layer.**

Database columns are `UUID NOT NULL`. The application generates the UUID v7 value before the INSERT statement. The database never calls `gen_random_uuid()` or any sequence function for PKs.

Idempotency keys for ledger entries are `TEXT` columns, not UUID v7. They are deterministically derived by the service layer from `(tenant_id, workflow_run_id, entry_type)` — e.g. a SHA-256 hex digest of those three fields. UUIDv7 is not deterministic; it cannot be used as a stable idempotency key. The `idempotency_key` column is separate from the row's UUID v7 PK.

---

## Consequences

**Enables:**
- Sequential insert performance on high-volume tables — new rows always append near the end of the B-tree, eliminating page splits under load.
- Safe idempotent retries: the caller generates the PK before the first attempt; on retry, the same key produces a 200 with the existing record rather than a duplicate row.
- Time-range queries on PKs are meaningful — `WHERE id > :last_seen_id` is an efficient pagination cursor.

**Forecloses:**
- Auto-increment integers as PKs — these would leak row counts (e.g., a tenant could infer how many `rating_decisions` exist by watching sequential IDs).
- Database-assigned PKs — FK references can only be built after a round-trip to the DB; UUID v7 removes this dependency.

**Migration note:** If an upstream table uses UUID v4 (e.g., Dify's existing `tenants` table), cross-table FK columns store the upstream type as-is. Only Credits Platform-owned PKs must be UUID v7.
